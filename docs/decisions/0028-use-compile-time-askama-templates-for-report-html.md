# 0028: Use Compile-Time Askama Templates For Report HTML

Date: 2026-07-20

Issues: [#241](https://github.com/rwjblue/antennabench/issues/241),
[#242](https://github.com/rwjblue/antennabench/issues/242)

## Status

Accepted.

Implemented by the report renderer migration tracked in #241. The completed
renderer streams compile-time fragments through the checked writer and retains
only the static CSS, generated geometry CSS, template adapter, and checked sink
as direct writer paths.

## Context

AntennaBench already separates report projection from presentation. The
renderer-neutral `SessionReport` owns the bounded facts, typed availability,
disclosure inputs, and chart data. Full and compact renderers turn that model
into deterministic standalone HTML with no scripts or external resources.
`CheckedHtmlWriter` enforces the local resource profile and cancellation
boundary while rendering.

The HTML implementation has nevertheless grown to roughly 5,500 lines. It
interleaves Rust control flow, report prose, table and SVG structure,
accessibility attributes, manual escaping, and hundreds of `write_html!` and
`push_str` calls. Long markup strings are difficult to read as HTML, tag and
attribute changes produce noisy Rust diffs, and every report-provided string
requires a local escaping decision. This was reasonable while the renderer was
small, but it now makes presentation work harder to review and increases the
risk that structurally similar full and compact views drift.

Changing the presentation mechanism must not move analysis into a template,
make generated reports depend on runtime files, weaken privacy boundaries, or
bypass byte and cancellation limits. The session bundle remains the durable
source of truth, `SessionReport` remains renderer-neutral, and generated HTML
remains a derived local artifact.

## Decision

### Compile-Time Fragment Templates

The report crate uses Askama templates compiled and embedded with the Rust
application. Templates live with the report crate and are not discovered,
loaded, or edited at runtime. AntennaBench does not expose user-provided report
templates, template scripting, development reload behavior, or a runtime
template registry.

Templates are section-sized fragments rather than one template that owns the
entire report pipeline. Rust continues to orchestrate:

- full versus compact output;
- ordered question families and goal-lens priority;
- audit and controller-evidence authorization;
- complete versus bounded detail; and
- the order in which fragments are streamed.

The orchestrator renders each fragment directly to the checked output sink. It
does not first render fragments into strings and then mark those strings as
safe inside a root template.

### Rust Presentation Models

Each migrated section receives a narrow Rust presentation model projected from
`SessionReport`. Rust presentation code continues to own:

- typed availability and empty-state selection;
- row membership, ordering, grouping, and omission;
- factual sentence and label selection;
- number, timestamp, unit, and enum formatting;
- chart scales, coordinates, geometry classes, map cells, and SVG points; and
- privacy, redaction, and disclosure policy.

Presentation models contain raw strings, numbers, booleans, collections, and
typed display variants. They do not contain pre-escaped strings or arbitrary
HTML. Askama owns markup, iteration, and simple display conditionals; it does
not recompute an estimand, interpret evidence, choose a favorable result, or
mutate report data.

Full and compact views share prepared facts when their evidence is the same.
They may use separate templates when compact omission or disclosure behavior
is intentionally different. Sharing a model must not cause audit-only data to
enter the compact template context.

### Escaping And Trusted Output

HTML auto-escaping is mandatory for every value supplied by a report or
presentation model. Report templates may not use Askama's `safe` filter,
disable auto-escaping, or accept a generic safe-HTML fragment. Template and
repository validation enforce this rule.

Static embedded CSS and Rust-generated geometry CSS remain an explicit trusted
writer path outside report-data templates. This narrow exception does not
permit report strings, controller output, operational history, imported data,
operator labels, or other model content to bypass auto-escaping.

Equivalent entity encodings or intentional template whitespace may change
generated bytes during migration. Legacy byte identity is not a compatibility
contract. Repeated rendering from the same report revision and options remains
byte-deterministic, and semantic structure, stable selectors, privacy, and
accessibility behavior remain unchanged unless a separate approved issue
changes them.

### Resource And Error Boundary

Askama renders incrementally through an adapter over `CheckedHtmlWriter` rather
than through an unrestricted intermediate `String`. The adapter stops template
rendering after a byte-limit or cancellation failure and preserves the first
typed `ReportResourceError`. Callers continue to receive no partial report.

An unexpected template-render failure is a typed report-render error, not a
panic and not a generic resource diagnostic. Compile-time template checking
eliminates ordinary syntax and field-reference failures from released builds;
runtime failures remain bounded and explicit.

### Incremental Migration

Renderer families migrate one at a time. A migration change introduces the
section view model and templates, switches full and compact call sites in its
scope, verifies the existing behavior contract, and removes the corresponding
legacy markup renderer. AntennaBench does not maintain two complete renderers
or postpone all switching to a flag day.

The final renderer contains no report-data-dependent `write_html!`, manual HTML
escaping, or direct markup `push_str` calls. Remaining direct writes are
limited to the checked sink and the documented static or generated CSS path.
An architecture test enumerates every report template, requires a corresponding
compiled Askama path, rejects escaping overrides and script markup, and checks
every remaining `push_str` site against that narrow allowlist.

## Consequences

Report structure becomes reviewable as HTML while Rust retains the type and
policy boundaries that make the report trustworthy. Template expressions and
model field references are checked during compilation, and escaping becomes a
default property of the rendering boundary rather than a convention repeated
at each interpolation site.

The migration adds Askama and its compile-time cost. Section presentation
models add code even when they closely resemble report-model rows, but that
code makes formatting, omission, and trust decisions explicit and testable.
Some canonical HTML bytes and string-fragment assertions will change; the
committed canonical sample must be regenerated whenever production output
changes.

Askama does not validate HTML semantics, accessibility, responsive layout,
print behavior, CSP embedding, or scientific meaning. Hostile-input tests,
typed resource-boundary tests, focused fragment snapshots, the canonical
sample check, and the real-browser embedded-report regression therefore remain
required.

## Rejected Alternatives

- Keeping Rust string assembly would avoid a dependency but would preserve the
  readability, escaping, and structural-review problems that motivated this
  decision.
- Handlebars is familiar and widely deployed, but its Rust implementation
  resolves a dynamic Serde value model through templates registered at
  runtime. AntennaBench owns its templates and benefits more from compile-time
  field checking than from cross-language or runtime template flexibility.
- MiniJinja and Tera provide capable runtime template environments, but dynamic
  loading and runtime expression resolution add flexibility this local,
  application-owned renderer does not require.
- Maud provides compile-time HTML generation and escaping, but its Rust-like
  markup DSL would keep presentation structure inside Rust source rather than
  create the desired HTML authoring boundary.
- Rendering child templates into strings and inserting them as safe HTML into
  a root template would simplify composition at the cost of a broad escaping
  exception and an intermediate allocation outside the checked writer path.
- A one-shot renderer rewrite would concentrate security, privacy, layout, and
  test churn in one change and would make behavioral drift difficult to
  isolate.
